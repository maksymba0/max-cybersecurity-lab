## Port Scanning Detection & TCP Flags

### Task 1: Observing a Port Scan from the Target Side

**Setup:**
- Target VM: `tcpdump` capture on `enp0s3`
- Attacker VM: `nmap 192.168.100.10`

**What I saw in tcpdump:**

IP 192.168.100.20.44824 > 192.168.100.10.80: Flags [S], seq 3491253645, win 64240
IP 192.168.100.20.60020 > 192.168.100.10.443: Flags [S], seq 1268321187, win 64240

**What the flags mean:**

| Flag | Name | Meaning |
|------|------|---------|
| `[S]` | SYN | Start connection (knock on door) |
| `[R]` | RST | Abort connection immediately (door slammed) |

**Why RST on closed ports:**  
The target responds with RST when no service is listening — meaning the port is closed.

### Task 2: SYN Scan (Stealth Scan)

**Command:**
```bash
sudo nmap -sS 192.168.100.10
```
**What -sS means**:
Part	Meaning
-s	Scan type
S	SYN scan (half-open)

|Scan	| Handshake| 	Logged?|	Stealth|
|------|------|---------|------------|
|-sT (TCP connect)	|Full (SYN → SYN-ACK → ACK)	|Yes	|Low|
|-sS (SYN stealth)	| Half-open (SYN → SYN-ACK → RST)	|Often not|High|


**What I Learned**:
- Closed ports respond with RST — open ports respond with SYN-ACK
- SYN scans are stealthier because they don't complete the handshake
- -sS is the standard nmap scan for a reason — it's faster and quieter
- Defenders can still detect SYN scans with tools like tcpdump or IDS rules
