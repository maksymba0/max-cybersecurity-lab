## SIEM Investigation: Brute Force Success

**Logs analyzed:**

2026-04-13 01:15:22 firewall src_ip=185.142.53.15 dst_ip=10.0.10.25 action=blocked
2026-04-13 01:15:25 web01 auth: Failed password for root from 185.142.53.15
2026-04-13 01:15:27 web01 auth: Failed password for root from 185.142.53.15
2026-04-13 01:15:30 web01 auth: Accepted password for root from 185.142.53.15


**What happened:**
- Attacker at IP 185.142.53.15 attempted to connect to server 10.0.10.25
- Firewall blocked the first connection attempt
- Attacker then tried SSH logins as root - two failures, then success
- Total time: 8 seconds from first failed to successful login

**Suspicion level:** HIGH
- 1 AM is abnormal for legitimate work
- Root account is high-value target
- Failed attempts followed by success = brute force

### Firewall Basics (What Actually Happened)

A firewall controls what traffic can enter or leave a network.

| Function | Meaning |
|----------|---------|
| **Block** | Prevents traffic from passing through |
| **Allow** | Permits specific traffic |
| **Log** | Records what was blocked or allowed |

In this case, the firewall **blocked** the attacker's first attempt - but the attacker kept trying and eventually got through via SSH (port 22). This means the firewall wasn't blocking port 22 entirely, or the attacker found another path.

**Why block port 22 (SSH) from the internet?**  
To prevent remote brute force attacks. If SSH is only accessible via VPN, external attackers can't even try.

**Firewall vs Application block:**

| Type | How It Works | Scope |
|------|--------------|-------|
| Firewall block | Prevents ANY connection to a port at network level | System-wide |
| Application block | The service itself refuses connections (e.g., disabling root login in SSH config) | Only affects that service |

Firewall blocking is stronger - it stops traffic before it reaches the service.

---

### Immediate response

1. Block IP `185.142.53.15` at the firewall
2. Change `root` password immediately
3. Audit logs for post-exploitation activity
4. Disable root SSH login permanently (`PermitRootLogin no` in `/etc/ssh/sshd_config`)

---

### Detection rule (for SIEM):
Alert when: (failed_logins > 3 in 60 seconds) AND (successful_login within 10 seconds)

### What I learned

- Firewall logs show initial blocking, but attackers can be persistent
- Successful brute force requires immediate containment
- Prevention (disabling root SSH, blocking port 22) is better than detection
- SOC analysts must act fast - every second counts
- Firewalls are the first line of defense; application config is the second
